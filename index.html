Section 01

What Is Full Stack Development?

Full Stack Development refers to the process of building complete web applications — both the visible frontend that users interact with, and the backend server logic and database that power it.

🖥️
Frontend
Everything a user sees and interacts with in a browser — built with HTML, CSS, and JavaScript.
⚙️
Backend
The server-side logic that processes requests, handles authentication, and communicates with the database.
🗄️
Database
Stores and manages all application data persistently — e.g. MySQL, PostgreSQL, MongoDB.
🔗
APIs
Connects frontend and backend. RESTful APIs allow data to be exchanged between layers securely.
📱
Responsive Design
Ensuring the application works beautifully on all screen sizes — mobile, tablet, and desktop.
🌐
Fetch & APIs
fetch() sends HTTP requests to external APIs. Use async/await to handle the response, response.json() to parse data, and try/catch for error handling.

🚀 Deployment & Hosting — The 4th Layer

A full stack application also needs a deployment layer — the infrastructure that makes your app available on the internet. Without it, your code only works on your local machine.

🌐
Web Server
Apache or Nginx listens for HTTP requests and serves your files or forwards requests to your backend app (e.g. Node.js, PHP).
☁️
Cloud Hosting
Platforms like Vercel, Netlify (frontend), Heroku, Render, or AWS host your app on servers accessible worldwide 24/7.
🔒
Domain & HTTPS
A custom domain name (e.g. rahulfoods.com) and an SSL certificate (HTTPS) encrypt traffic between the user's browser and your server.
Complete 4-Layer Stack: Frontend (HTML/CSS/JS) → Backend (Node.js / PHP) → Database (MySQL / MongoDB) → Deployment (Web Server + Cloud Hosting + Domain). A full stack developer understands and can work across all four layers.

🍔 Real-World Example: Food Delivery

Step 1 — Frontend (Customer): The customer visits the website, browses the menu, selects items, and clicks "Order Now". JavaScript validates the form and sends the order to the server.
Step 2 — Backend (Server): PHP/Python receives the HTTP request, verifies the customer is logged in, calculates the total, processes the payment gateway, and prepares an order object.
Step 3 — Database (Storage): MySQL saves the order with details: customer ID, items, price, timestamp, and delivery address. The server sends back a confirmation response.
Step 4 — Response: JavaScript receives the confirmation and shows the customer "Your order #1042 is confirmed! Arriving in 30 minutes." — all without reloading the page.

⚙️ Backend Code — PHP

PHP runs on the server. It receives the order from the frontend, saves it to the database, and sends back a JSON response.

server.php
// Receive order data sent from JavaScript fetch() $data = json_decode(file_get_contents('php://input'), true); $item = $data['item']; $price = $data['price']; // Save order into MySQL database $stmt = $pdo->prepare( "INSERT INTO orders (item, price) VALUES (?, ?)" ); $stmt->execute([$item, $price]); // Send JSON success response back to browser echo json_encode([ 'status' => 'success', 'message' => 'Order saved!' ]);

🟢 Backend Code — Node.js + MongoDB

Node.js with Express and MongoDB (a NoSQL database) stores orders as JSON documents — no fixed table structure required.

server.js (Node.js + Express + MongoDB)
// Import Express and MongoDB client (Mongoose ODM) const express = require('express'); const mongoose = require('mongoose'); const app = express(); app.use(express.json()); // Connect to MongoDB Atlas (cloud database) mongoose.connect('mongodb+srv://user:pass@cluster.mongodb.net/foodapp'); // Define a MongoDB schema (structure of each document) const Order = mongoose.model('Order', { item: String, price: Number, createdAt: { type: Date, default: Date.now } }); // POST /api/order — saves order as a MongoDB document app.post('/api/order', async (req, res) => { const order = new Order(req.body); await order.save(); // inserts into MongoDB res.json({ status: 'success', id: order._id }); }); app.listen(3000);
MySQL vs MongoDB: MySQL stores data in rows and tables (relational). MongoDB stores data as flexible JSON-like documents — ideal for rapidly changing data structures like food orders with varying options.

💡 HTML Structure Explained

DOCTYPE Declaration: <!DOCTYPE html> tells the browser this is an HTML5 document. Always placed at the very top.
Semantic HTML: Using meaningful tags like <header>, <nav>, <main>, <footer> instead of generic <div> improves accessibility and SEO.
Attributes: Tags can have attributes like class, id, href, src, and alt which provide extra information or functionality.
Nesting: HTML elements can be nested inside each other to create structure. Proper nesting is essential for valid, readable HTML code.
ARIA Attributes: Accessible Rich Internet Applications attributes like aria-label, aria-hidden, and role make pages accessible to screen readers and assistive technologies. Example: <button aria-label="Close menu">.

The .topics-grid class uses CSS Grid with grid-template-columns: repeat(3, 1fr) to create three equal columns. Resize the browser to see it collapse to 2 columns on tablet and 1 column on mobile.

/* ── Desktop (> 768px): 3 equal columns ── */ .topics-grid { display: grid; grid-template-columns: repeat(3, 1fr); gap: 1.5rem; } /* ── Tablet (576px – 768px): 2 columns ── */ @media (max-width: 768px) { .topics-grid { grid-template-columns: 1fr 1fr; } } /* ── Mobile (< 480px): 1 column (stacked) ── */ @media (max-width: 480px) { .topics-grid { grid-template-columns: 1fr; } }

🎛️ CSS Custom Properties (Variables)

CSS variables store reusable values. Define them once in :root and use them anywhere — change one value to update your whole theme.

style.css — Variables
/* Define custom properties on :root (global scope) */ :root { --primary-color: #00d4ff; --secondary-color: #7c3aed; --font-size-base: 1rem; --border-radius: 12px; --spacing-lg: 2rem; } /* Use them with var() */ .btn { background: var(--primary-color); border-radius: var(--border-radius); padding: var(--spacing-lg); } /* Dark mode: just redefine the same variables */ .dark-mode { --primary-color: #7c3aed; /* overrides root */ }
Why use variables? This portfolio uses --accent, --bg-primary, --text-secondary etc. as CSS variables. The entire dark/light theme switch works by redefining these variables on body.light-mode — no JS colour changes needed.

🖱️ CSS Pseudo-classes & Pseudo-elements

Pseudo-classes target elements in a specific state. Pseudo-elements style a specific part of an element — all without extra HTML.

style.css — Pseudo-classes
/* :hover — triggered when mouse is over element */ .btn:hover { background: #0099cc; transform: scale(1.05); } /* :focus — triggered when element is focused (keyboard/click) */ input:focus { outline: 2px solid var(--primary-color); border-color: var(--primary-color); } /* :nth-child() — select specific children */ tr:nth-child(even) { background: rgba(0,0,0,0.05); } li:first-child { font-weight: bold; } li:last-child { border-bottom: none; } /* ::before / ::after — insert decorative content */ .section-title::before { content: '// '; color: var(--primary-color); }

Bootstrap Framework

Bootstrap is the world's most popular open-source CSS framework. It provides ready-to-use components and a powerful grid system that makes building responsive websites much faster.

⚖️ Bootstrap: Advantages vs Limitations

A side-by-side comparison to understand when and why to use Bootstrap:

Feature ✅ Advantage ⚠️ Limitation
Speed Pre-built components save hours of coding from scratch Full Bootstrap CSS adds extra file size (~22 KB minified + gzip)
Responsiveness Mobile-first design is built in by default — works on all screen sizes Custom breakpoints require overriding default Sass variables
Consistency Uniform appearance across all modern browsers automatically Sites can look similar to other Bootstrap sites without customisation
Components Buttons, navbars, modals, forms, cards — all ready to drop in Overriding default component styles requires high-specificity CSS
Customisation Easy to add custom CSS on top; full Sass variable access Class names are verbose and can be hard to memorise at first
Updates Large community, well-maintained, detailed official documentation Major version upgrades (v4 → v5) can break existing projects

➡️ Arrow Functions & Template Literals (ES6)

ES6 (2015) introduced cleaner, modern JavaScript syntax. Arrow functions and template literals are now standard in all modern JS code.

main.js — ES6 Syntax
// ── Traditional function vs Arrow function ────────── // Old way: function keyword function greet(name) { return "Hello, " + name + "!"; } // ES6 Arrow function — shorter syntax const greet = (name) => `Hello, ${name}!`; // Single param: no brackets needed const double = n => n * 2; // No params: empty brackets required const sayHi = () => "Hi!"; // ── Template Literals ─────────────────────────────── const name = "Rahul"; const course = "CET138"; const score = 88; // Old string concatenation const msg1 = "Hello " + name + ", your score in " + course + " is " + score; // Template literal — backticks + ${expression} const msg2 = `Hello ${name}, your score in ${course} is ${score}`; // Multi-line strings (impossible with old quotes) const html = ` <div class="card"> <h2>${name}</h2> <p>Score: ${score * 1.2}</p> </div> `; // Arrow functions in array methods const nums = [1, 2, 3, 4, 5]; const doubled = nums.map(n => n * 2); // [2,4,6,8,10] const evens = nums.filter(n => n % 2 === 0); // [2,4] const total = nums.reduce((a, b) => a + b, 0);// 15

💾 localStorage & Event Delegation

localStorage stores data in the browser that persists after the page closes. Event delegation attaches one listener to a parent instead of many to children.

localStorage — Save & Load Data
// localStorage only stores strings — use JSON for objects // SAVE — store user theme preference localStorage.setItem('theme', 'dark'); // LOAD — read on page load const theme = localStorage.getItem('theme'); // 'dark' // SAVE object — must stringify first const user = { name: "Rahul", score: 88 }; localStorage.setItem('user', JSON.stringify(user)); // LOAD object — must parse back const saved = JSON.parse(localStorage.getItem('user')); console.log(saved.name); // "Rahul" // DELETE one key localStorage.removeItem('theme'); // CLEAR all localStorage localStorage.clear();
Event Delegation — one listener handles all children
// ❌ Bad — separate listener per button (inefficient) document.querySelectorAll('.menu-item button') .forEach(btn => btn.addEventListener('click', handleClick)); // ✅ Good — ONE listener on the parent (event delegation) document.getElementById('menu-grid') .addEventListener('click', (e) => { // e.target is the element actually clicked if (e.target.classList.contains('add-to-cart')) { const name = e.target.dataset.name; const price = e.target.dataset.price; addToCart(name, price); } }); // Works even for buttons added AFTER page load!
Live demo: The shopping cart in this portfolio uses event delegation — clicking any "Add to Cart" button is caught by one listener on the menu grid parent div, not by individual listeners on each button.
